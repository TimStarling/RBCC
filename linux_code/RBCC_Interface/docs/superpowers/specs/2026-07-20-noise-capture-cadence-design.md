# 噪声采集 0.8 秒、4.5 秒更新周期设计

## 背景

`RBCC_Interface/noise_worker.py` 当前在单独进程中连续运行。每轮采集 2.0 秒音频，完成分析与频率选择后立即开始下一轮，因此正常更新间隔约为 2.1 秒。`main.py` 中的 4000 ms 定时器只检查工作进程是否存活，不控制采集周期。

## 目标

- 每轮音频采集时长固定为 0.8 秒。
- 相邻两轮采集的开始时间正常情况下相隔 4.5 秒。
- 一轮采集与计算耗时小于 4.5 秒时，等待本周期剩余时间；按当前约 11 ms 的计算耗时，典型等待约为 3.7 秒。
- 停止信号在采集和周期等待期间都能及时终止工作进程。
- 界面显示的采集时长与实际配置一致。

## 不在范围内

- 不改变噪声分析、最佳频率评分或串口协议。
- 不改变 `main.py` 中每 4 秒一次的工作进程健康检查。
- 不解决当前系统缺少物理声卡、录音源为全零的问题。

## 方案

在 `noise_worker.py` 中定义采集时长与更新周期常量。每轮开始时记录 `time.monotonic()`，完成采集、分析、结果输出或错误输出后，计算本轮已经消耗的时间，并通过 `stop_event.wait()` 等待 `max(0, 4.5 - 已耗时)`。

此方案使用“采集开始到下一次采集开始”的固定周期。等待使用单调时钟，避免系统时间调整影响节拍；使用 `stop_event.wait()` 而不是 `time.sleep()`，保证停止信号可以立即打断等待。如果一轮耗时超过 4.5 秒，不启动重叠采集，而是在该轮结束后立即开始下一轮。

## 代码变更

1. `noise_worker.py`
   - 文档说明从连续 2 秒采集改为 0.8 秒采集、4.5 秒周期。
   - 增加 `CAPTURE_DURATION_SECONDS = 0.8` 与 `UPDATE_INTERVAL_SECONDS = 4.5`。
   - `AlarmConfig.analysis_duration` 使用采集时长常量。
   - 增加纯计算函数，用于根据周期开始时间和当前单调时间计算剩余等待秒数。
   - 正常与异常路径统一在每轮末尾等待剩余周期；移除异常路径中固定 2 秒等待，避免重复等待。
2. `main.py`
   - 启动状态文字更新为“正在采集 0.8 秒音频…”。

## 错误处理

- 无输入设备、录音失败或分析失败时继续输出 `no_input` 事件。
- 错误轮次也遵循 4.5 秒起始间隔，防止失败时高频重试。
- 收到 SIGTERM 或 SIGINT 后，采集函数和周期等待都通过同一个 `stop_event` 退出。

## 测试与验收

- 先增加失败测试，验证采集时长常量为 0.8、更新周期常量为 4.5。
- 单元测试验证：一轮耗时 0.8 秒时剩余等待为 3.7 秒；一轮耗时超过 4.5 秒时等待为 0。
- 运行现有噪声项目测试，确认分析与频率选择未回归。
- 在香橙派上启动实际 worker，记录至少三次 `captured_at`，确认相邻更新间隔接近 4.5 秒。
- 检查界面源码和运行状态文字不再显示“采集 2 秒”。

## 成功标准

正常运行时每轮采集 0.8 秒，相邻采集开始时间为 4.5 秒节拍；不产生并行或重叠采集，停止操作保持可中断，现有频率计算逻辑不变。
