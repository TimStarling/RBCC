# 自动采集与串口蜂鸣播放设计

## 背景

`RBCC_Interface/noise_worker.py` 已自动执行 0.8 秒噪声采集，并按 4.5 秒的采集开始间隔计算最佳警告频率。worker 会把 `best_alarm_hz` 发送给界面主进程，但主进程目前只显示该频率，没有通过串口触发蜂鸣器。

香橙派当前仅有 PulseAudio `auto_null` 空输出，没有可实际发声的物理音频输出。因此本功能不调用 `play_audio()`，而是使用已经连接的 `/dev/ttyUSB0` 控制外部蜂鸣器。

## 目标

- 无需手动操作，界面启动后自动采集、计算并发送最佳频率。
- 每轮采集 0.8 秒，相邻两轮采集开始时间正常情况下相隔 4.5 秒。
- 只选择评分最高的一个频率，候选范围严格限制在 2000–5000 Hz。
- 频率转换为 2000–5000 范围内的四位整数。
- 通过现有 `/dev/ttyUSB0`、115200 8N1 串口发送精确 UTF-8 数据 `beep:XXXX`。
- 数据不包含引号、空格、回车或换行。例如 3000 Hz 对应 9 字节 `beep:3000`，十六进制为 `626565703a33303030`。

## 架构

worker 保持只负责采集与算法计算，使用 `AlarmConfig` 将警告频率范围设为 2000–5000 Hz、数量设为 1。worker 继续通过 JSON 行事件向主界面输出 `best_alarm_hz`。

主界面已经持有并轮询唯一的 `serial.Serial` 实例。收到 `status=ok` 的噪声事件后，主界面验证并编码最佳频率，然后复用这个实例写入 `beep:XXXX` 并调用 `flush()`。这样不会让 worker 重复打开 CH340 串口，也不会干扰现有传感器读取。

## 数据流

1. 界面自动启动 noise worker。
2. worker 采集 0.8 秒音频。
3. worker 在 2000–5000 Hz 内选择最高评分的一个频率。
4. worker 输出包含 `best_alarm_hz` 的 JSON 事件。
5. 主界面显示最佳频率，并将其编码成 UTF-8 `beep:XXXX`。
6. 主界面通过已连接的 `/dev/ttyUSB0` 写入 9 字节命令，外部蜂鸣器播放。
7. 下一轮仍按 4.5 秒采集起始节拍执行。

## 校验与错误处理

- 编码函数拒绝非有限数值、非四位整数以及超出 2000–5000 的频率。
- 串口未连接时继续显示计算结果，不让采集循环崩溃；自动连接恢复后，下一轮结果会正常发送。
- 串口写入不足 9 字节、`OSError` 或 `serial.SerialException` 会显示发送失败状态，但不会终止 worker。
- 播放命令与传感器读取复用同一个串口对象，不创建第二个串口连接。
- 不添加自动重发队列，避免连接恢复时播放过期频率。

## 测试与验收

- 先写失败测试，验证 worker 配置的范围为 2000–5000、频率数量为 1。
- 先写失败测试，验证 3000 编码为精确字节 `b"beep:3000"`，边界 2000 和 5000 可用，越界值被拒绝。
- 使用记录写入内容的假串口验证写入 9 字节并调用 `flush()`；此处模拟硬件边界，不模拟算法。
- 验证主界面收到 `status=ok` 后调用串口发送逻辑；串口断开或写入失败时界面不崩溃。
- 运行现有 4 项采集周期测试与 26 项噪声算法回归测试。
- 在香橙派 `/dev/ttyUSB0` 上执行一次受控发送，验证主机侧返回 9/9 字节写入，并记录 UTF-8 十六进制。

## 不在范围内

- 不使用香橙派的空音频输出播放 PCM。
- 不改变传感器串口帧解析、TCP 转发或摄像头功能。
- 不改变 0.8 秒采集时长和 4.5 秒采集开始间隔。
